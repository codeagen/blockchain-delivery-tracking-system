Blockchain Delivery Management — UI

Rough wireframes mapping the design space for a decentralised delivery system. Three roles, seven core screens, explored through four different navigation patterns. Structure & flow first — not final pixels.

Roles: Customer · Seller · Delivery Agent Stack ref: React · NestJS · Ethereum / Solidity Lifecycle: create → assign → dispatch → transit → confirm
Status palette Dispatched In transit Delivered On-chain / verified
A · Persistent Sidebar
For: Sellers & operators on desktop who live in the app all day.
A fixed left rail holds every destination; a role badge sits up top. Best for power users managing many deliveries, where the on-chain ledger deserves a permanent home in the nav.
+ everything one click away + room for ledger + filters – heavy for mobile / casual customers

B · Top Tab Bar
For: A lean, focused desktop console — fewer destinations, kept horizontal.
Primary sections live as tabs across the top; content fills the width below. Reads like a familiar web app and keeps notifications + tracking a single jump away without a heavy rail.
+ simple, scannable, full-width content + low nav overhead – limited room as destinations grow

C · Mobile Bottom-Nav
For: Customers tracking a package and agents working in the field — thumb-first.
A fixed bottom bar with 4–5 destinations; one screen at a time, big tap targets. This is where proof-of-delivery confirmation and live tracking truly belong.
+ thumb-reachable, native feel + ideal for customers & agents – cramped for seller bulk ops

D · Role Hub / Launcher
For: One adaptive entry point that reshapes itself per role.
A home "hub" of large action tiles tailored to who signed in; everything else is a focused screen you drill into and back out of. Lowest learning curve across all three roles — the nav is the task list.
+ one design serves all roles + gentle for first-time users – more taps to reach deep screens
